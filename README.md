# utils

## 简介

`utils` 是一个面向六合彩相关开发场景的通用工具集合，聚焦于项目开发过程中高频出现的基础能力建设，包括数据处理、号码规则封装、日志调试、配置管理、请求封装、测试辅助以及工程化最佳实践等内容。  
在实际业务中，很多团队都会反复编写类似的辅助代码，既影响开发效率，也容易因为实现不统一而增加维护成本。`utils` 的目标，就是将这些常见能力沉淀为可复用、可扩展、可协作的工具模块，帮助开发者更专注于核心业务逻辑本身。 🚀

无论你是在搭建数据采集脚本、开发号码分析功能、维护后台服务，还是整理前后端共用的规则函数，这个仓库都可以作为一个稳定的基础工具层来使用。它不仅关注“能不能用”，也强调“是否清晰、是否易测、是否便于长期维护”。

---

## 特点

- **聚焦六合彩开发场景**：围绕常见业务需求设计工具函数与模块结构，减少重复造轮子。
- **模块化组织**：工具按职责拆分，便于按需引用、独立维护与持续扩展。
- **统一编码风格**：推荐清晰命名、单一职责、可测试实现，降低团队协作成本。
- **支持配置化开发**：将环境变量、业务参数和规则映射解耦，适合多环境部署。
- **便于测试与调试**：内置或推荐日志、Mock、断言等实践，提升问题定位效率。
- **兼容工程化流程**：适合集成 ESLint、Prettier、CI、单元测试等常见开发工具链。
- **强调可维护性**：优先提供稳定、可复用、边界明确的工具设计思路。 ✅

## 快速开始

下面给出一个推荐的项目接入方式，你可以根据自己的技术栈进行调整。

### 1. 克隆仓库

```bash
git clone https://github.com/your-org/utils.git
cd utils
```

### 2. 安装依赖

如果项目使用 Node.js 生态，可以执行：

```bash
npm install
```

或：

```bash
pnpm install
```

### 3. 启动开发环境

```bash
npm run dev
```

### 4. 运行测试

```bash
npm test
```

### 5. 构建产物

```bash
npm run build
```

如果你只希望把其中某些通用模块集成到现有项目中，也可以直接复制 `src/utils` 下的工具文件，或将其封装为内部私有包供多个项目共享。

## 使用说明

`utils` 更适合作为“基础能力层”接入你的业务仓库，而不是直接承载完整业务。推荐按以下方式使用：

### 1. 按领域划分工具模块

建议将工具划分为几类：

- `number/`：号码格式化、校验、排序、映射等
- `date/`：日期处理、期号相关辅助逻辑
- `request/`：接口请求封装、重试机制、超时处理
- `logger/`：开发日志、错误输出、调试追踪
- `config/`：环境配置读取与校验
- `test/`：测试数据构造、Mock 方法、断言辅助

这样的拆分方式有利于后续演进，避免工具函数散落在各个业务目录中。

### 2. 保持函数“纯”与“短”

工具函数尽量做到：

- 输入清晰
- 输出稳定
- 副作用可控
- 命名语义明确

例如，不建议在一个函数里同时完成“请求接口 + 数据转换 + 日志上报 + 页面提示”四件事，而应拆成多个职责单一的小模块，便于复用和测试。

### 3. 统一错误处理策略

在六合彩相关系统中，数据来源、接口稳定性、规则变更频率都可能影响运行结果。因此建议：

- 对外部请求增加超时与重试
- 对关键字段进行空值判断
- 对异常使用统一错误对象或统一日志出口
- 在生产环境中保留必要的可追踪信息

### 4. 使用配置驱动规则

不要把常量、映射关系、环境开关直接硬编码在业务逻辑中。推荐将常用规则抽离到配置层，例如：

```js
export const appConfig = {
  timeout: 5000,
  retryCount: 2,
  enableDebug: true,
};
```

这样在规则调整或环境切换时，维护成本会显著降低。

### 5. 配合规范化开发流程

建议在仓库中加入以下约束：

- `ESLint`：保证代码风格一致
- `Prettier`：减少格式争议
- `Jest` / `Vitest`：覆盖核心工具函数测试
- `Husky` + `lint-staged`：在提交前执行检查
- GitHub Actions：自动化测试与构建

这些并不是“锦上添花”，而是长期维护工具库时非常关键的基础设施。 🛠️
